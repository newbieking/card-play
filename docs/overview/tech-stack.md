# 卡牌冒险养成游戏 — 技术选型与架构方案

> 配套文档：`play.md`（核心玩法框架）、`design.md`（美术素材规范）
> 目标平台：**移动（iOS/Android）+ PC（Windows/Mac）+ H5/微信小游戏，一套代码**
> 本文范围：技术栈选型论证 + 客户端/服务端/数据/美术管线整体架构 + 核心系统技术落地方案（不写代码）

---

## 一、技术栈选型与论证

### 1.1 客户端引擎选型

| 候选方案 | 跨平台覆盖 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| **Cocos Creator 3.x (TS)** | H5/微信小游戏原生；PC 走 Web/Electron；移动端一键打包 iOS/Android | 真正一套代码多端；小游戏/国内渠道适配最成熟；TS 开发快；包体小 | 3D 表现力弱于 Unity（本项目为 2D 卡牌，无影响） | ✅ **推荐** |
| Unity (C#) | 全平台强；H5 需 WebGL/小游戏适配层 | 生态/插件最丰富，3D 强 | H5/小游戏需额外适配层、包体大、TS 团队需切 C# | ❌ 备选 |
| 纯前端 (React/Vue + PixiJS/Phaser) | H5+PC 原生；移动端需 Capacitor 打包 | 前端栈熟悉、Web 生态好 | 移动端原生能力/性能/打包体验弱，工程化需自建 | ❌ 备选 |

**结论**：采用 **Cocos Creator 3.x + TypeScript**。2D 卡牌游戏无需 Unity 的 3D 能力，而 Cocos 在 H5/微信小游戏/移动端的"一套代码"体验最佳，且 TS 与后端同语言，降低协作成本。

### 1.2 服务端技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 语言/运行时 | **Node.js (TypeScript)** | 与客户端同语言，开发快，复用类型定义与配置表解析逻辑 |
| Web 框架 | NestJS 或 Fastify | 模块化、依赖注入（NestJS）或轻量高吞吐（Fastify） |
| 关系库 | PostgreSQL | 玩家档案、卡牌、订单等需要事务/强一致的永久数据 |
| 缓存/排行 | Redis | 在线态、会话、爬塔/竞技场实时排行榜（Sorted Set） |
| 通信协议 | WebSocket（长连）+ Protobuf/JSON | 战斗/实时交互走 WS；配置下发可用 HTTP |
| 消息队列 | Redis Stream / BullMQ | 异步任务（邮件发放、付费到账、战斗回放） |
| 部署 | Docker + K8s / 轻量云 | 无状态逻辑服水平扩容，网关前置负载均衡 |

> 选型理由：服务端逻辑以"配置驱动的数值结算 + 状态管理"为主，而非高并发计算密集，Node.js(TS) 完全胜任，且与 Cocos 客户端共享 TS 类型与配置表 schema，可显著减少联调成本。

### 1.3 美术资源管线（衔接 `design.md`）

- **卡牌插画**：PNG 透明底，按稀有度分目录（common/rare/epic/legendary）
- **特效/VFX**：PNG 序列帧（小游戏兼容）或 WebM（H5/PC），保留透明通道
- **图标/UI**：SVG 优先（矢量缩放），导出 PNG 位图供 Cocos 使用
- **卡背/场景**：PNG，可含轻微纹理
- **资源管理**：Cocos Asset Bundle 分包（基础包 + 按章节/活动远程加载），支持热更

## 二、整体架构

### 2.1 客户端分层

```
┌─────────────────────────────────────────────┐
│  表现层 (View)    UI 界面 / 卡牌动画 / 特效播放 │
├─────────────────────────────────────────────┤
│  战斗表现层 (BattleView)  回合制演出、特效驱动  │
├─────────────────────────────────────────────┤
│  业务逻辑层 (Logic)  玩法流程、状态机、本地校验 │
├─────────────────────────────────────────────┤
│  数据层 (Data)  配置表(CSV/JSON)、本地存档缓存 │
├─────────────────────────────────────────────┤
│  网络层 (Net)  WS 封装、断线重连、协议编解码    │
└─────────────────────────────────────────────┘
```

### 2.2 服务端分层

```
客户端
  │  WebSocket / HTTP
  ▼
┌──────────────┐    ┌──────────────────────────────┐
│  网关 Gateway │───▶│  逻辑服 Logic（按模块微服务化）│
│ 连接/鉴权/限流│    │ Account/Card/Gacha/Battle/...  │
└──────────────┘    └───────────┬──────────────────┘
                                │
                ┌───────────────┼───────────────┐
                ▼               ▼               ▼
          ┌──────────┐   ┌──────────┐    ┌──────────┐
          │PostgreSQL│   │  Redis   │    │ MQ(异步) │
          │ 永久数据  │   │缓存/排行  │    │邮件/订单 │
          └──────────┘   └──────────┘    └──────────┘
```

### 2.3 数据驱动设计

- 所有数值（卡牌属性、技能倍率、升星消耗、装备强化曲线、抽卡概率）以 **CSV/JSON 配置表** 管理
- 配置表经构建工具生成 TS 类型 + 二进制/JSON 资源，客户端与服务端**共用同一份 schema**
- 策划用 Excel 编辑 → 导出 CSV → 自动生成类型与校验 → 热更下发，无需改代码

## 三、服务端各模块详细设计

### 3.1 接入与基础设施层

| 模块 | 职责 |
|---|---|
| **网关 Gateway** | 连接管理、WS/HTTP 接入、协议编解码（Protobuf）、心跳、断线重连、限流防刷、鉴权透传 |
| **配置中心 Config Center** | 配置表版本管理、按需/全量下发、热更推送、客户端版本校验 |
| **数据持久化** | PostgreSQL（玩家档案/永久数据，事务保证一致性）；Redis（在线态/会话/排行榜/缓存） |
| **消息队列 MQ** | 异步解耦：邮件发放、订单到账、战斗回放、活动结算 |

### 3.2 业务功能模块

1. **账号与登录 Account**
   - 游客登录 → 微信/Apple/Google 第三方绑定
   - 设备指纹 + Token 鉴权；防多开、防账号共享
   - 数据落 PostgreSQL `player_account`

2. **玩家档案 Profile**
   - 等级、资源（金币/钻石/卡牌碎片/经验药水/装备材料/升星石）、背包
   - 资源变更统一走 **经济服务接口**，保证原子性（避免并发超发）

3. **卡牌模块 Card**
   - 收藏图鉴（已获得/未获得，点亮奖励）
   - 升星（1★→10★ 阶梯消耗校验）、合成（同名同稀有度 +1 星，安全合成仅扣材料）
   - 技能树升级（Lv1-Lv10，按卡牌升星解锁槽位），技能消耗资源分怒气/冷却两种
   - 重复卡牌转化碎片（按稀有度差异化：普通×5 / 稀有×15 / 史诗×30 / 传说×80）
   - **面板属性计算**（运行时，不落库）：HP/ATK/MATK/DEF/MDEF/SPD 五维，由 `card_def` + `card_stat_growth` + `card_star_up` 公式计算
   - **战力计算**：`(HP×0.5 + ATK + MATK + DEF×0.8 + MDEF×0.8) × (1 + SPD×0.01)`
   - 升星后同步更新 PVP 防守阵容快照 `defense_power`
   - 详见 `data-model.md §3.3`（实例表）、`progression-tables.md`（数值表）、`combat-system.md §1.2`（面板公式）

4. **抽卡模块 Gacha** ⭐
   - 卡池类型：基础 / 职业限定 / 传说精华 / 限时（概率 UP）
   - **保底逻辑全部在服务端**：十连必出稀有以上、英雄召唤必出传说、限时池独立保底计数
   - 随机数使用**服务端安全随机源**（加密级 RNG + 种子），不下发客户端
   - 概率公示数据独立存储，满足合规审计

5. **战斗校验模块 Battle** ⭐
   - 自动/半自动回合制：客户端上传**阵容（2 前排 + 3 后排）+ 操作指令序列**，服务端以确定性战斗帧重演
   - **混合资源机制**：主动技能（slot 1）消耗怒气（普攻+20/受击+10/击杀+30，上限 100），终极技能（slot 3）走冷却制（4-8 回合）
   - **面板分离**：物理职业走 ATK+DEF，法系职业走 MATK+MDEF，双曲线伤害公式（K=500）
   - 服务端权威结算战力/伤害/胜负，**校验客户端上报结果**，防篡改
   - 战斗记录可回放（存 MQ/对象存储），用于客服与反作弊
   - 详见 `combat-system.md`（规则）、`battle-protocol.md`（协议）

6. **冒险关卡 Adventure**
   - 章节/关卡进度（每章 10-20 关，末关 Boss）、解锁状态机
   - 掉落结算（金币/经验药水/碎片/装备材料），关卡增益减益规则（如"火系 +20%"）
   - 关卡首通/星级记录

7. **爬塔模块 Tower**
   - 无尽层数进度（每 10 层 Boss，其余随机敌人，战力递增 + 特殊规则叠加）
   - 每 5 层"命运选择"三选一 Buff、连续通关翻倍
   - 每日 3 次免费刷新楼层奖励；全球/好友排名写入 Redis

8. **装备模块 Equipment**
   - 武器/护甲/饰品强化（+1~+15，高等级失败需保护符）
   - 精炼（追加随机属性）、套装激活（2/4/6 件套）
   - 强化结果服务端结算，失败不扣卡牌仅扣材料

9. **经济与商城 Economy**
   - 商店兑换（碎片/代币）、充值（月卡/战令/直购/礼包）
   - **支付回执验签**（平台回调 → 服务端校验 → 发奖），防伪造订单
   - 付费点配置化（价格区间、限购、周期）

10. **竞技场 PVP Arena** ⭐
    - **异步战报模式**：攻击方提交阵容+指令 vs 防守方 AI 操作，服务端权威结算
    - **战力匹配**：按防守方阵容战力 ±5%~20% 递进搜索匹配对手
    - **积分制段位**：青铜→白银→黄金→铂金→钻石→大师→传说（7 段位），4 周一赛季
    - 防守阵容保存时快照（`defense_snapshot`），战斗中使用快照属性
    - 防守方获得 HP+10% 补偿 AI 劣势
    - 连胜 5+ 额外 +5 积分/场；大段位不掉段
    - 详见 `pvp-arena.md`（完整规则）、`data-model.md §3.11`（数据库表）

11. **排行榜 Ranking**
    - 爬塔/竞技场：Redis Sorted Set 实时排名，支持全球榜 + 好友榜
    - PVP 排行榜按段位积分（score）排序
    - 榜单快照定时归档（周榜重置、赛季结算）

11. **公会 Guild**
    - 公会创建/加入、公会副本、公会战、公会贡献与商店

12. **任务与日常 Quest**
    - 日常 9 项 + 周常 8 项任务、活跃度宝箱（30/60/100 三档）
    - 体力系统：120 上限 / 6 分钟自然恢复 / 午晚餐补给 / 递增购买
    - 体力参数存于 `global_const` 配置表，运行时读取
    - 登录签到（7 日循环 + 累计里程碑）
    - 新手 7 日破冰计划
    - 详见 `quest-system.md`、`data-model.md §3.12`（quest_progress 表）

13. **活动 Activity**
    - 限时卡池、限时活动代币、世界 Boss / 周 Boss 团本（多人协作结算）

14. **邮件与补偿 Mail**
    - 系统邮件、补偿/奖励异步发放队列（走 MQ），附件领取校验

15. **好友社交 Friend**
    - 好友列表、赠送/切磋、助力（如爬塔协助）

### 3.3 横切关注点

| 关注点 | 方案 |
|---|---|
| **防作弊** | 战斗服务端校验、抽卡服务端随机、关键数值（如真实战力/掉落）不下发客户端 |
| **日志监控** | 行为日志、付费埋点、服务性能/错误监控（Prometheus + 日志平台） |
| **GM 后台** | 发放/封禁/数据查询/配置热更开关 |
| **版本兼容** | 协议版本号 + 配置版本号，客户端过期强制更新，热更资源灰度 |

## 四、核心系统技术落地要点

### 4.1 自动/半自动回合制战斗
- 5 张卡牌统一阵型（2 前排 + 3 后排），阵容部署在客户端完成，提交 `BattleStart` 协议
- 战斗过程由**确定性指令序列**驱动，服务端以相同种子 + 相同逻辑重演，输出权威结果
- **混合资源机制**：主动技能消耗怒气（客户端可发 Action 触发），终极技能走冷却（同理）
- 半自动：玩家在 30 秒操作时限内发送主动/终极技能 Action，超时服务端代填默认行动
- 详见 `battle-protocol.md`（协议定义）、`combat-system.md`（伤害公式与规则）

### 4.2 抽卡保底
- 每个卡池维护玩家独立保底计数器（存 PostgreSQL `gacha_pity_counter`，按 player_id + pool_id）
- 抽卡请求 → 服务端 RNG → 命中保底则强制提升稀有度 → 写审计日志 `gacha_audit`
- 概率配置从 Config Center 下发前端展示，但**判定在服务端**（双端同源防不一致）
- 详见 `gacha-rng.md`（算法实现）、`config-tables.md §3.8/3.9`（配置 Schema）

### 4.3 养成数值（升星/技能/装备）
- 全部消耗与产出走配置表；数值变更经经济服务原子接口
- 升星阶梯（1★→10★）、技能解锁（随星级）、装备强化曲线均数据驱动，策划可调
- 面板属性运行时计算（ATK/MATK/DEF/MDEF 五维分离），不冗余落库
- 升星后同步更新 PVP 防守阵容快照 `defense_power`（data-model.md §3.11）
- 详见 `progression-tables.md`（数值表）、`data-model.md §3.3/3.4`（实例表）

### 4.4 PVP 竞技场
- **异步战报模式**：攻击方提交阵容+指令 vs 防守方 AI 自动操作，服务端权威结算
- **战力匹配**：按防守方阵容战力 ±5%~20% 递进搜索，3-15 秒内匹配
- **积分制段位**：4 周一赛季，积分软重置，大段位不掉段
- 防守阵容保存即快照（`defense_snapshot`），战斗中使用快照属性，防止"换装偷鸡"
- 详见 `pvp-arena.md`（完整规则）、`data-model.md §3.11`（arena_data 表）

### 4.5 爬塔无尽 + 排行榜
- 层数进度持久化；每 5 层命运选择为服务端下发的随机三选一（服务端随机源）
- 排行榜实时写入 Redis Sorted Set，读取走缓存，周/赛季重置用定时任务归档
- PVP 排行榜按段位积分（score）排序（`arena:rank:season:{id}`）

### 4.5 资源热更新
- Cocos Asset Bundle 分包：基础包 + 章节/活动远程包
- 服务端 Config Center 推版本号，客户端比对后增量下载，无需发版更新内容

| 阶段 | 目标 | 关键交付 |

## 六、选型小结

- **客户端**：Cocos Creator 3.x (TypeScript) —— 一套代码覆盖移动/PC/H5/小游戏
- **服务端**：Node.js (TypeScript) + NestJS/Fastify + PostgreSQL + Redis —— 与客户端同语言，配置表 schema 复用
- **数据驱动**：21 张 CSV/JSON 配置表 + 自动生成 TS 类型，策划热更无需改码
- **美术管线**：衔接 `design.md` 的 PNG/WebM/SVG 规范，Cocos Asset Bundle 分包热更
- **安全基线**：战斗/抽卡服务端权威校验，支付回执验签，关键数值不下发客户端
